OneEHR

Real TJH data, end-to-end modeling, held-out testing, unified evaluation, and a polished web console in one demo.

01

What This Demo Shows

  • Real dataset: TJH train cohort plus external held-out cohort.
  • Full chain: preprocess → train → test → analyze → eval build → eval run → eval report.
  • Production-style UX: one web UI for model results, test audit, comparison, and eval-first experiment review.
  • LLM integration: Zenmux OpenAI-compatible backend with openai/gpt-5.4.

System View

OneEHR Tutorial Pipeline

The demo is structured so every stage leaves durable artifacts that feed the next stage and stay explorable in the UI.

02
Stage 1

Preprocess

Standardize TJH raw Excel into train and external-test CSV tables.

Stage 2

Train

Fit tabular baselines on the train cohort with durable manifests and split outputs.

Stage 3

Test

Run held-out evaluation on the external cohort and persist test summaries.

Stage 4

Analyze

Emit dataset, cohort, prediction, test, temporal, and interpretability modules.

Stage 5

Eval Build

Freeze patient-level evaluation instances and evidence bundles from the saved run.

Stage 6

Eval Run

Call Zenmux-backed framework systems and align them against the same frozen evidence.

Stage 7

Eval Report

Write leaderboard, paired deltas, and compare-run summaries in one reproducible artifact tree.

Every stage stays queryable from the web UI and the CLI.

Why This Matters

  • Held-out testing is explicit, not buried inside a notebook cell.
  • Frozen eval instances force every framework to see the same saved evidence.
  • Leaderboard artifacts make model-vs-framework comparisons reproducible.
  • Compare-run summaries make baseline-vs-showcase deltas demoable.

Demo Message

OneEHR is not only about training models. It is about scoring every system on a shared contract that humans can inspect.
This is the bridge from research pipeline to defensible, agent-era evaluation.
